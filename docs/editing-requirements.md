# 网页编辑功能需求分析与分解

> 基于竞品调研、社区反馈及技术可行性调查，为 go-grip 项目定义网页编辑功能的完整需求。
> 本文档仅描述「做什么」，不涉及「怎么做」，不包含任何代码、库名、协议或架构选择。

---

## 一、需求来源与定位

### 1.1 为什么需要这个功能

在同赛道 CLI Markdown 预览工具（Python grip、vmd、Marked 2、mdp 等）中，无一提供网页编辑能力——此功能是 go-grip 的差异化优势，让用户在「看到渲染结果后想立刻改」的场景下无需切换回外部编辑器。

### 1.2 解决谁的什么痛点

| 用户画像 | 使用频率 | 核心痛点 | 编辑场景 |
|---|---|---|---|
| **重度用户**（开源维护者、文档工程师） | 每天 2-5 次 | 侧边栏浏览发现排版/措辞问题时，需切换回编辑器修改，每次上下文切换耗时 3-5 秒 | 在预览页面就地修正排版错误、更新链接、微调措辞 |
| **中度用户**（技术文档作者） | 每周数次 | 关注格式一致性（表格对齐、列表缩进），在编辑器和预览之间反复比对效率低下 | 渲染后即时发现格式问题并当场修正 |
| **轻度用户**（个人开发者） | 偶尔使用 | 写完文档预览时发现错别字、失效链接等小错误，需回到编辑器找文件、修改、再刷新 | 看到错别字或链接问题，一键切换到编辑态快速修正 |

### 1.3 边界声明

本功能**不做**以下事情：

- **不做**全功能 Markdown 编辑器或 IDE —— 对标 GitHub 的「Edit / Preview」标签切换体验，解决高频微编辑场景
- **不做**实时协同编辑、多人协作、冲突合并
- **不做**文件新建、删除、重命名等文件管理操作
- **不做**图片拖拽上传、附件管理
- **不做**版本历史、undo 回退到任意历史版本
- **不做**Markdown 语法自动补全、格式化、Lint 检查
- **不做**移动端深度适配 —— 编辑功能以桌面端为主要目标，移动端保证基本可用即可

## 二、功能需求 (FR)

### 2.1 编辑入口

- **FR-01** 用户在预览页面上应能看到一个含义清晰无歧义的编辑入口控件（如按钮或图标）。【P0：MVP 必须】
- **FR-02** 用户激活编辑入口后，页面应从预览模式切换为编辑模式。【P0：MVP 必须】
- **FR-03** 编辑入口仅当当前页面展示的是 Markdown 文件（`.md` 扩展名）时可见；在非 Markdown 文件或空状态页面、以及用户切换到侧边栏其他文件后，编辑入口的指向应随之更新。【P0：MVP 必须】

### 2.2 编辑体验

- **FR-04** 进入编辑模式时，编辑器应自动填入当前 Markdown 文件的原始文本内容（未渲染的 Markdown 源码），而非渲染后的 HTML。【P0：MVP 必须】
- **FR-05** 编辑区域应占据主要内容区的全高，使用户能舒适地浏览和修改 Markdown 内容。【P0：MVP 必须】
- **FR-06** 编辑模式下的视觉呈现应与预览模式有明显区别（如不同的背景色、边框或布局），使用户清楚感知当前处于编辑状态而非查看状态。【P0：MVP 必须】
- **FR-07** 编辑器应支持标准文本编辑操作：光标移动、文本选择、剪切/复制/粘贴、撤销/重做。【P0：MVP 必须】
- **FR-08** 编辑器的配色方案应跟随当前主题（深色/浅色）自动切换，确保编辑体验与页面整体风格一致。【P1：第二阶段】
- **FR-09** 编辑区域应使用等宽字体显示 Markdown 源码，保持代码块、表格等内容的对齐效果。【P1：第二阶段】

### 2.3 保存行为

- **FR-10** 用户在编辑模式下应能通过一个明确的「保存」操作，将编辑器中的完整内容覆盖写入磁盘上的原始文件（全量保存，非增量或行级保存）。【P0：MVP 必须】
- **FR-11** 系统应支持通过键盘快捷键（Ctrl+S 或 Cmd+S）触发保存，该快捷键在编辑器获得焦点时生效，且不触发浏览器默认的「保存网页」行为。【P1：第二阶段】
- **FR-12** 保存操作完成后，系统应向用户给出明确的成功反馈（如短暂提示消息），反馈应在 1-3 秒内自动消失。随后预览区域应自动刷新并展示最新渲染结果，页面从编辑模式退回预览模式。【P0：MVP 必须】
- **FR-13** 预览刷新后，用户当前的浏览位置应尽可能恢复（如滚动到编辑前的位置附近），以提供连贯的阅读体验。【P1：第二阶段】

### 2.4 取消编辑

- **FR-14** 用户在编辑模式下应能通过一个明确的「取消」操作退出编辑状态，编辑器中的所有未保存修改应被丢弃，页面恢复到进入编辑模式前的预览状态。【P0：MVP 必须】
- **FR-15** 用户执行取消操作或尝试关闭浏览器标签页时，如果编辑器内容与原始文件存在差异，系统应弹出确认提示，要求用户明确确认放弃修改后再离开。【P1：第二阶段】

### 2.5 文件操作范围

- **FR-16** 编辑功能仅适用于 Markdown 文件（扩展名为 `.md`），读取文件的原始内容（非渲染后的 HTML）进行编辑。【P0：MVP 必须】

## 三、非功能需求 (NFR)

### 3.1 安全性

- **NFR-01** 系统必须防止路径穿越攻击：任何文件操作的路径必须经过规范化校验，确保目标文件严格位于被服务目录的范围之内。【P0：MVP 必须】
- **NFR-02** 系统必须将可保存的文件类型限制为 Markdown 文件（`.md` 扩展名），拒绝保存其他文件类型。【P0：MVP 必须】
- **NFR-03** 系统必须限制单次保存操作的内容大小上限，防止超大内容导致服务端资源耗尽。【P0：MVP 必须】
- **NFR-04** 文件写入操作必须是原子性的：保存过程中如发生意外中断，原始文件不应被部分写入或损坏。【P0：MVP 必须】
- **NFR-05** 系统必须在并发场景下保证同一文件的写入安全，不发生数据竞争或交错写入。【P1：第二阶段】

### 3.2 兼容性

- **NFR-06** 编辑功能必须不干扰现有热重载机制：文件保存后，所有已连接的浏览器标签页应自动刷新，与通过外部编辑器修改文件的行为一致。【P0：MVP 必须】
- **NFR-07** 编辑功能必须不影响现有键盘导航（左右箭头键切换文章）：编辑模式下键盘导航自动禁用。【P0：MVP 必须】
- **NFR-08** 编辑功能必须不影响深色/浅色主题切换，且编辑功能引入的所有视觉元素须使用独立命名前缀，不与现有样式和脚本发生冲突。【P0：MVP 必须】

### 3.3 可用性

- **NFR-09** 编辑界面必须在桌面级屏幕（宽度 >= 1024px）上提供舒适的完整编辑体验；平板级屏幕（>= 768px）上保持合理可用，不出现布局断裂。【P0：MVP 必须】
- **NFR-10** 所有错误提示信息必须使用用户可理解的表述，禁止暴露服务端内部路径、堆栈信息或技术细节。【P1：第二阶段】

## 四、异常场景与边界 (EC)

### 4.1 编辑过程中

- **EC-01** 当前文件为空文件（0 字节）时，编辑入口应正常可用，编辑器打开空白区域供用户输入并保存。【P0：MVP 必须】
- **EC-02** 当前文件内容很大（如数万行）时，系统应能正常加载；若超过系统配置的大小上限，应在进入编辑前告知用户，而非在保存时才报错。【P1：第二阶段】
- **EC-03** 用户处于编辑模式时，若该文件被外部程序修改导致热重载触发，编辑器应保持打开状态且不丢失用户正在编辑的内容。【P1：第二阶段】

### 4.2 保存过程中

- **EC-04** 保存请求因网络问题（连接断开、超时等）失败时，系统应明确告知用户「保存失败」，保留编辑器内容不丢失，允许重试。【P0：MVP 必须】
- **EC-05** 保存请求因服务端错误（磁盘已满、权限不足、路径无效等）失败时，系统应向用户展示清晰可理解的错误原因，保留编辑器内容不丢失。【P0：MVP 必须】
- **EC-06** 当原始文件已被外部进程删除时，用户保存应收到文件不存在的提示并建议退出编辑模式。【P1：第二阶段】

### 4.3 多用户与并发场景

- **EC-07** 同一文件在多个标签页或浏览器中同时编辑并保存时，采用后写者胜出策略，不提供合并或冲突检测。并发保存不产生数据损坏。【P2：未来考虑】

## 五、服务模式差异 (SM)

go-grip 存在两种服务模式：单文件模式（指定单个 .md 文件）和目录模式（指定目录，服务其中所有 .md 文件）。

### 5.1 单文件模式

- **SM-01** 编辑入口和编辑功能应完全可用，编辑操作作用于命令行指定的唯一 Markdown 文件；不存在文件切换场景，故不触发切换文件时的未保存提醒。【P0：MVP 必须】

### 5.2 目录模式

- **SM-02** 编辑入口应作用于当前侧边栏中高亮选中的文件，而非固定文件。【P0：MVP 必须】
- **SM-03** 用户处于编辑状态时点击侧边栏其他文件，系统应检测未保存修改并提示用户选择「保存并切换」「放弃并切换」或「继续编辑」。【P1：第二阶段】
- **SM-04** 保存后预览刷新应保持侧边栏的展开/折叠状态、当前文章高亮及 TOC 可见状态不变。【P0：MVP 必须】

## 六、接受标准 (AC)

### 6.1 MVP 接受标准（对应 P0）

- **AC-01** 核心编辑闭环：用户在目录模式下预览任意 .md 文件 → 看到编辑入口 → 点击进入编辑模式（已预填原始内容）→ 修改 → 保存 → 预览区自动刷新展示最新渲染结果。单文件模式下同样可用。
- **AC-02** 安全检查到位：拒绝非 .md 文件的保存请求；拒绝路径穿越请求；超限内容被拒绝且不执行写入。原子写入保证文件不损坏。
- **AC-03** 兼容性零退化：编辑功能引入后，热重载、键盘导航、主题切换、侧边栏展开折叠均不受影响。
- **AC-04** 异常可恢复：网络故障或服务端异常时，用户看到清晰错误提示，编辑器内容不丢失，可重试保存。
- **AC-05** 桌面浏览器兼容：Chrome、Firefox、Edge 最近两个大版本上，编辑入口、界面布局、保存/取消操作全部正常。

### 6.2 完整版接受标准（对应 P1+）

- **AC-06** 快捷键与提示增强：Ctrl+S/Cmd+S 保存、Esc 取消（需确认）；保存成功后短暂成功提示后自动退回预览模式；未保存修改离开时弹出确认对话框。
- **AC-07** 边界场景覆盖：空文件可编辑并保存；外部删除文件时保存给出明确提示并建议退出；大文件加载前预判并提示；编辑期外部修改不丢失编辑器内容。
- **AC-08** 主题与响应式：编辑器配色跟随主题自动同步；平板设备（>= 768px）上布局正常；所有错误提示语言清晰无技术细节泄露。
