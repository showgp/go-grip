# 编辑模式分屏实时预览需求分析

> 为 go-grip 已有的网页编辑功能添加分屏实时渲染预览。本文档仅描述「做什么」，不涉及「怎么做」，不包含任何代码、库名、协议或架构选择。
> 基于 `editing-requirements.md`（基础编辑需求）和 `editing-implementation-plan.md`（实现规划）之上构建。

---

## 一、背景与定位

### 1.1 现状

go-grip 当前已实现完整的网页编辑闭环（编辑 > 保存 > 刷新 > 预览）。编辑器工具栏有三按钮：`Preview`、`Save`、`Cancel`。当前 `Preview` 和 `Cancel` 行为完全相同——均退出编辑模式并丢弃修改。用户在编辑过程中完全看不到渲染结果，只能凭 Markdown 源码判断格式是否正确。

### 1.2 目标

在编辑模式下引入分屏布局：左侧为可编辑的 textarea（Markdown 源码），右侧为实时渲染的 Markdown 预览。预览由浏览器端轻量级 Markdown 解析器驱动，在用户输入时实时更新。最终保存仍走服务端 goldmark 完整渲染管线。

### 1.3 边界声明

- **不做** goldmark 扩展的完整浏览器端复刻（CJK emphasis、mermaid、mathjax、attributes、filename headers、emoji、高亮等）。浏览器端预览仅覆盖 GFM 基础语法 + 部分常用扩展，作为快速编辑辅助而非最终渲染。
- **不做** 所见即所得（WYSIWYG）编辑器——始终保留 Markdown 源码编辑 + 预览的二分模式。
- **不做** 预览面板内的交互（链接点击、代码复制等），预览仅用于视觉参照。
- **不做** 将分屏预览作为页面默认浏览模式——仅在编辑模式下激活。

## 二、浏览器端 Markdown 解析器候选调研

### 2.1 约束条件

- 体积须极小（优先 <10KB min+gzip，最大 <30KB min+gzip）
- 必须支持 GitHub Flavored Markdown 核心语法：表格、删除线、任务列表、围栏代码块、自动链接
- 须能以静态文件形式嵌入 go-grip 项目（放入 `defaults/static/js/` 或通过构建时下载嵌入）
- 完整复刻 goldmark 自定义扩展（CJK emphasis、mermaid、mathjax、attributes、filename headers、emoji、高亮）**不作要求**——这些由最终服务端 goldmark 渲染承担

### 2.2 候选对比

| 项目 | 原始体积 | Gzip 体积 | GFM 支持 | 扩展性 | 许可证 | CDN 可用 | 活跃维护 |
|------|---------|----------|---------|--------|--------|---------|---------|
| **marked** | ~40KB | ~12KB | 完整（表格、删除线、任务列表、围栏代码块、自动链接） | 丰富（自定义渲染器/分词器/钩子） | MIT | jsdelivr / unpkg | 是（v18+, 2025 持续更新） |
| **snarkdown** | ~2KB | ~1KB | 无（无表格、无任务列表、无自动链接、无删除线） | 无 | MIT | jsdelivr / unpkg | 否（最后更新 2018） |
| **markdown-it** | ~124KB | ~45KB | 需插件组合（基础核心不含 GFM，需 markdown-it-table + markdown-it-task-lists 等） | 极强（插件生态） | MIT | jsdelivr | 是（v14+, 2025 持续更新） |
| **micromark** | ~15-30KB (含必要扩展) | ~5-10KB | 需扩展（micromark-extension-gfm 系列包） | 低层 API | MIT | jsdelivr | 是（v4+, 2025 持续更新） |
| **zero-md** | ~80KB+ (含 marked 依赖) | ~30KB+ | 继承 marked | 无（Web Component 封装） | ISC | jsdelivr / unpkg | 是 |

### 2.3 重点候选分析

#### marked (~40KB raw / ~12KB gzip)

**优势**：
- GFM 语法全部内置，开箱即用（`marked.parse(content)` 一行调用）
- API 简洁稳定，用于预览场景无需深入定制
- MIT 许可证友好
- 广泛使用（npm 周下载量数千万），社区成熟
- 支持 `marked.use({ extensions })` 轻量扩展，未来如需添加 emoji 或自定义渲染可渐进增强
- 提供预构建的 UMD/ESM 产物，可直接复制为静态文件

**劣势**：
- 12KB gzip 略超 10KB 理想上限，但仍在 30KB 上限内
- 默认不清理输出 HTML（需开发者确保 textarea 输入的 HTML 标签被转义或预览内容仅从 textarea.value 渲染 Markdown 源码，天然避免 XSS）

#### snarkdown (~2KB raw / ~1KB gzip)

**优势**：极致体积。

**劣势**：完全不支持 GFM。无表格、无任务列表、无自动链接、无删除线、无 HTML 块。放弃。

#### markdown-it (~124KB raw / ~45KB gzip)

**优势**：插件生态极强，可精确组合所需语法。

**劣势**：45KB gzip 远超 30KB 上限。核心不含 GFM，需额外插件增加体积。放弃。

#### micromark (~15-30KB raw, 含 GFM 扩展 / ~5-10KB gzip)

**优势**：体积可控，符合 CommonMark 规范，低层控制力强。

**劣势**：需要手写扩展组合代码（micromark-extension-gfm-table + gfm-task-list-item + gfm-strikethrough + gfm-autolink-literal 等），集成复杂度高于 marked。对于"快速编辑辅助预览"场景过度工程化。

### 2.4 推荐

**推荐使用 marked**。理由：

1. GFM 全部内置，一行调用完成渲染——对预览场景足够
2. ~12KB gzip 在 30KB 硬上限内，是最小的具备完整 GFM 支持的选择
3. 无需管理多个扩展包、无需手写扩展组合逻辑
4. 未来如需添加 emoji 或基础 CJK 支持可通过 `marked.use()` 渐进增强
5. 广泛使用、持续维护、MIT 许可证

## 三、分屏预览功能需求 (FR-SPLIT)

### 3.1 分屏布局

- **FR-SPLIT-01** 编辑模式下，系统应在主内容区呈现左右分屏布局：左侧为可编辑的 textarea（Markdown 源码），右侧为实时渲染的 Markdown 预览面板。【P0：MVP 必须】
- **FR-SPLIT-02** 左右分屏比例应可调整（如通过拖拽分隔条），默认比例为 1:1（各占 50%）。【P1：第二阶段】
- **FR-SPLIT-03** 分屏布局在视口宽度 >= 1024px 时必须正常工作（桌面端核心体验）；768px-1023px 之间保持可用（允许缩小比例但不可断裂）；<768px 时预览面板应堆叠到编辑器下方而非侧边。【P0：MVP 必须】
- **FR-SPLIT-04** 预览面板应可被用户收起/展开（通过工具栏按钮或分隔条控件），收起时编辑器占据全宽（恢复到当前"仅编辑"模式），展开时恢复分屏。【P0：MVP 必须】
- **FR-SPLIT-05** 编辑模式进入时预览面板默认展开（即分屏为编辑模式的默认状态）。【P0：MVP 必须】

### 3.2 预览面板内容渲染

- **FR-SPLIT-06** 预览面板应使用浏览器端 Markdown 解析器将 textarea 内的 Markdown 源码实时转换为 HTML 并渲染。【P0：MVP 必须】
- **FR-SPLIT-07** 预览面板应至少支持以下 Markdown 语法：标题（1-6 级）、粗体/斜体、删除线、链接、图片、无序/有序列表、任务列表（含勾选状态）、表格（含对齐）、围栏代码块（含语言标签）、内联代码、引用块、分割线、自动链接、段落与换行。【P0：MVP 必须】
- **FR-SPLIT-08** 预览面板内容与最终服务端 goldmark 渲染结果之间允许存在视觉差异（CJK emphasis、mermaid 图、mathjax 公式、emoji 短码、alert/ghissue/details 块等不做浏览器端支持）。预览为编辑辅助，最终渲染以保存后服务端输出为准。【P0：MVP 必须】
- **FR-SPLIT-09** 预览面板不响应链接点击、不执行代码复制、不初始化 mermaid 渲染——仅作为静态的样式化 HTML 呈现。【P0：MVP 必须】

### 3.3 预览面板样式

- **FR-SPLIT-10** 预览面板的 Markdown 渲染样式应复用现有的 GitHub 风格 Markdown CSS（`github-markdown-light.css` / `github-markdown-dark.css`），确保预览结果与最终渲染在基础排版上视觉一致。【P0：MVP 必须】
- **FR-SPLIT-11** 预览面板的字体、配色方案应自动跟随当前页面主题（浅色/深色 / `prefers-color-scheme` 媒体查询）切换。【P0：MVP 必须】
- **FR-SPLIT-12** 预览面板应有独立的垂直滚动条，与编辑器滚动条分离，允许用户独立浏览编辑区和预览区。【P0：MVP 必须】
- **FR-SPLIT-13** 预览面板在内容为空时应显示占位提示（如"预览将显示在此处"或等效的视觉占位），而非完全空白。【P2：未来考虑】

### 3.4 与现有编辑 UI 的共存

- **FR-SPLIT-14** 编辑模式下，侧边栏、TOC、页面导航、footer 的隐藏行为（当前通过 `[data-editing="true"]` CSS 控制）必须保持与现有行为一致——分屏模式下这些元素继续隐藏。【P0：MVP 必须】
- **FR-SPLIT-15** `.preview-content` div（包含服务端渲染的最终输出）在编辑模式下必须保持隐藏，不与新的实时预览面板同时可见。【P0：MVP 必须】
- **FR-SPLIT-16** 实时预览面板的 DOM 结构应使用独立于 `.preview-content` 的类名和容器，避免与现有服务端渲染内容产生 CSS 或 JS 选择器冲突。【P0：MVP 必须】

## 四、实时渲染策略需求 (FR-LIVE)

### 4.1 渲染触发策略

- **FR-LIVE-01** 预览面板应在用户停止输入后自动更新渲染（防抖策略），而非每次击键都触发渲染。【P0：MVP 必须】
- **FR-LIVE-02** 防抖延迟应在用户感知上足够即时（建议 150-300ms），使用户感觉预览是"实时"的。【P0：MVP 必须】
- **FR-LIVE-03** 用户可通过工具栏按钮手动触发即时刷新（如"刷新预览"按钮或在预览面板收起后重新展开时自动刷新）。【P2：未来考虑】

### 4.2 大文档处理

- **FR-LIVE-04** 当文档行数超过 5000 行时，系统应自动将防抖延迟延长或切换为仅在用户停止输入一段时间后渲染，防止频繁渲染导致浏览器卡顿。【P1：第二阶段】
- **FR-LIVE-05** 系统应在渲染操作执行前检查 Markdown 内容长度，超过阈值（如 100KB）时提示"文档较大，预览性能可能受影响"，但不阻止渲染或编辑。【P2：未来考虑】
- **FR-LIVE-06** 渲染操作应是异步非阻塞的，不得阻塞 textarea 的输入响应——用户在大文档中继续输入时不应感到延迟或卡顿。【P0：MVP 必须】

### 4.3 滚动同步（P2）

- **FR-LIVE-07** 编辑区域与预览面板之间可选支持滚动同步：当用户在 editor 中滚动时，预览面板自动滚动到对应渲染区域附近（nice-to-have）。【P2：未来考虑】
- **FR-LIVE-08** 滚动同步仅作为可选项提供（工具栏开关），且不强制要求精确行级对应——大致段落级对应即可。【P2：未来考虑】

## 五、与现有编辑流程的集成需求 (FR-INT)

### 5.1 Preview 按钮行为变更

- **FR-INT-01** 编辑工具栏中的 `Preview` 按钮行为应从"等同于 Cancel"变更为"切换预览面板的显示/隐藏"——即切换分屏模式与纯编辑器模式。【P0：MVP 必须】
- **FR-INT-02** `Preview` 按钮在分屏激活时应有视觉状态标识（如高亮或按下态），表示预览面板当前可见；在纯编辑器模式下取消该标识。【P0：MVP 必须】
- **FR-INT-03** 点击 `Preview` 按钮切换预览面板时，编辑器内容不得丢失，编辑状态（光标位置、选中范围、未保存标记）不得改变。【P0：MVP 必须】

### 5.2 进入/退出编辑模式

- **FR-INT-04** `enterEditMode()` 进入编辑模式后，预览面板应默认显示（即默认分屏状态），`Preview` 按钮初始状态应为"激活/高亮"。【P0：MVP 必须】
- **FR-INT-05** `exitEditMode()` 退出编辑模式时，预览面板内容应被清空或隐藏，释放其所占用的 DOM 和内存资源。【P0：MVP 必须】
- **FR-INT-06** 退出编辑模式时，页面应恢复到编辑前的预览状态（`.preview-content` 再次可见），不保留分屏布局的任何 DOM 残留。【P0：MVP 必须】

### 5.3 toggleUI 函数适配

- **FR-INT-07** `toggleUI()` 函数在进入编辑模式（editMode=true）时，除现有行为（显示 editor-toolbar / editor-container、隐藏 preview-toolbar / preview-content）外，还应显示预览面板容器并开始渲染。【P0：MVP 必须】
- **FR-INT-08** `toggleUI()` 函数中分屏预览面板的显示/隐藏逻辑应独立于 `Preview` 按钮切换——即 `toggleUI(true)` 始终进入编辑模式（带分屏），`Preview` 按钮仅在编辑模式内切换预览面板可见性。【P0：MVP 必须】

### 5.4 键盘快捷键

- **FR-INT-09** 编辑模式下现有的 `Ctrl+S` / `Cmd+S` 保存、`Esc` 取消、`Ctrl+Enter` 保存并停留的行为保持不变，不与分屏预览引入的新功能冲突。【P0：MVP 必须】
- **FR-INT-10** 可选新增 `Ctrl+P` 快捷键用于切换预览面板显示/隐藏（等价于点击 Preview 按钮），该快捷键仅在 textarea 获得焦点时生效。【P2：未来考虑】

### 5.5 侧边栏切换与未保存提示

- **FR-INT-11** 分屏模式下，用户通过侧边栏点击切换到其他文件时，现有的三选项对话框（保存并切换 / 放弃并切换 / 继续编辑）行为保持不变。【P0：MVP 必须】
- **FR-INT-12** 分屏模式下，`beforeunload` 拦截未保存修改的行为保持不变。【P0：MVP 必须】

## 六、构建与分发需求 (FR-DIST)

### 6.1 分发策略

- **FR-DIST-01** 浏览器端 Markdown 解析器必须作为静态文件随 go-grip 内置分发，不得依赖运行时从 CDN 远程加载。【P0：MVP 必须】
- **FR-DIST-02** 解析器静态文件应放置在 `defaults/static/js/` 目录下，与现有 JS 文件（`editor.js`、`theme-switch.js` 等）处于同一层级，通过 Go 的 `embed.FS` 机制嵌入二进制。【P0：MVP 必须】
- **FR-DIST-03** 解析器 JS 文件应在 `layout.html` 的 `<head>` 中通过 `<script>` 标签引用，引用位置在 `<script src="/static/js/editor.js" defer>` 之前，确保编辑器脚本执行时解析器全局符号已可用。【P0：MVP 必须】

### 6.2 推荐理由

选择静态文件嵌入而非 CDN 运行时加载，理由如下：

1. **离线可用**：go-grip 是本地开发工具，用户可能在无网络环境（飞机、离线开发、内部网络）使用。CDN 依赖会导致预览功能在这些场景下完全失效
2. **性能确定**：静态本地文件无 DNS 解析、TLS 握手、CDN 回源延迟，首字节时间确定且最小
3. **版本锁定**：静态文件提交到仓库后版本确定，不受 CDN 上游版本变更或废弃影响
4. **安全合规**：不向第三方 CDN 发送请求，不引入外部依赖审计负担
5. **体积可控**：目标解析器（~12KB gzip）作为静态文件对 go-grip 二进制体积影响约 40KB（未压缩），符合 go-grip 的轻量定位

## 七、非功能需求 (NFR-SPLIT)

### 7.1 性能

- **NFR-SPLIT-01** 浏览器端 Markdown 解析器初始加载（parse + eval）应不阻塞页面首次渲染。通过 `<script defer>` 或等效机制确保主内容优先呈现。【P0：MVP 必须】
- **NFR-SPLIT-02** 解析器渲染操作在 1000 行以内的文档上应在 16ms 内完成（不丢帧），在 5000 行文档上应在 100ms 内完成。【P1：第二阶段】
- **NFR-SPLIT-03** 存在分屏预览面板的情况下，编辑器输入延迟（从击键到字符出现在 textarea）不应超越浏览器 textarea 原生性能——渲染操作不得阻塞主线程输入处理。【P0：MVP 必须】

### 7.2 兼容性

- **NFR-SPLIT-04** 分屏预览功能必须在 Chrome、Firefox、Edge 最近两个大版本上正常工作。【P0：MVP 必须】
- **NFR-SPLIT-05** 分屏预览功能不得干扰现有热重载机制（保存后所有标签页自动刷新）。【P0：MVP 必须】
- **NFR-SPLIT-06** 分屏预览功能不得干扰现有主题切换机制（浅色/深色 / 系统媒体查询）。【P0：MVP 必须】
- **NFR-SPLIT-07** 分屏预览引入的 CSS 选择器和 JS 变量必须使用独立命名前缀，不与 `editor.css`、`docs-layout.css` 及 `server.go` 中的模板变量产生冲突。【P0：MVP 必须】

### 7.3 安全

- **NFR-SPLIT-08** 预览面板中渲染的 HTML 内容必须避免 XSS 风险。鉴于内容来源是用户自己在 textarea 中输入的 Markdown 源码（非不可信外部输入），且解析器将 Markdown 转换为 HTML 后直接插入预览 DOM，如使用 `innerHTML` 设置内容必须确保解析器本身不引入 XSS 向量（如不渲染 `<script>` 标签、不执行事件处理器）。【P0：MVP 必须】

### 7.4 可维护性

- **NFR-SPLIT-09** 分屏预览的 JS 逻辑应在 `editor.js` 中实现（新增函数模块），不创建独立的第二个 JS 文件，保持编辑功能的前端代码集中在单一文件中。【P0：MVP 必须】
- **NFR-SPLIT-10** Markdown 解析器静态文件（第三方库）与 go-grip 自身 JS 代码（`editor.js` 等）必须明确区分——第三方库文件仅包含解析器原始发布代码，不做 go-grip 特定修改。如需定制解析行为，应在 `editor.js` 中通过解析器提供的扩展 API 实现。【P0：MVP 必须】

## 九、接受标准 (AC-SPLIT)

### 9.1 MVP 接受标准（对应 P0）

- **AC-SPLIT-01** 分屏编辑闭环：用户点击 Edit 进入编辑模式 > 默认看到左侧编辑器 + 右侧实时预览 > 在左侧输入 Markdown 文本 > 右侧在停止输入后自动更新渲染 > 点击 Save 保存 > 页面刷新展示服务端 goldmark 最终渲染结果。
- **AC-SPLIT-02** Preview 按钮行为正确：编辑模式下点击 Preview 按钮可切换预览面板的显示/隐藏；按钮状态反映当前面板可见性；切换时编辑器内容不丢失。
- **AC-SPLIT-03** 响应式分屏：桌面端 (>=1024px) 左右分屏正常；平板端 (768-1023px) 分屏可用不断裂；移动端 (<768px) 预览堆叠在编辑器下方。
- **AC-SPLIT-04** 主题兼容：预览面板配色跟随浅色/深色/系统媒体查询自动切换，与页面整体视觉一致。
- **AC-SPLIT-05** 零退化：现有编辑功能（保存、取消、未保存提示、侧边栏切换、热重载、键盘快捷键）全部正常，不因分屏引入而退化。
- **AC-SPLIT-06** 离线可用：浏览器端 Markdown 解析器作为静态内嵌文件加载，无需外部网络请求即可完成实时预览。
- **AC-SPLIT-07** XSS 防护验证：在 textarea 中输入 `<script>alert(1)</script>`，预览面板不得执行该脚本；输入 `<img src=x onerror=alert(1)>`，事件处理器不得执行。注意：marked.js 默认透传 raw HTML，此为本地开发工具的 self-XSS 边界，可接受但此行为需在实现文档中记录。
- **AC-SPLIT-08** 异步渲染不阻塞输入：编辑 2000+ 行文档时，在 textarea 中键入字符应感觉响应迅速（击键与字符出现在屏幕之间无感知延迟）。渲染操作异步执行，不阻塞主线程的输入处理。
- **AC-SPLIT-09** 独立滚动：编辑器和预览面板各自拥有独立的垂直滚动条，互不影响。

### 9.2 增强接受标准（对应 P1+）

- **AC-SPLIT-10** 分屏比例可拖拽调整：用户可通过拖拽中间分隔条调整左右面板比例。
- **AC-SPLIT-11** 大文档性能：5000+ 行文档输入时编辑器不卡顿，渲染延迟自动调节。
